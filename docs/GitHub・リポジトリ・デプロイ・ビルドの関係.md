# GitHub・リポジトリ・デプロイ・ビルドの関係（図解・比喩）

用語どうしのつながりを、比喩と図で説明します。

---

## 1. 比喩：工場と配送で考える

| 用語 | 比喩での意味 |
|------|--------------|
| **GitHub** | **倉庫の会社**。「コードの保管場所」を提供している。 |
| **リポジトリ（repo）** | 倉庫のなかの**ひとつの部屋・箱**。1つのプロジェクト（ここでは daily-mood-checkin）のコードが入っている。 |
| **push（プッシュ）** | あなたのPCから **「その箱にコードを届ける」** こと。＝ コードを GitHub のリポジトリに送る。 |
| **ビルド** | 届いた材料（コード）を **「動く形に組み立てる」** こと。ここでは `pip install` でライブラリを入れて、実行できる状態にする。 |
| **デプロイ** | 組み立てたものを **「本番の場所に置いて、動かし始める」** こと。＝ Render のサーバーに置いて、URL で誰でもアクセスできるようにする。 |

**流れの比喩**  
あなたが「箱」に設計図（コード）を届ける（push）→  Render がその箱を取りに来て、中身を組み立て（ビルド）、お店に並べて開店（デプロイ）。

---

## 2. 図：全体の流れ

```
  【あなたのPC】                     【GitHub】                    【Render】
  コードを書く                       倉庫                          本番の場所

       │                                │                              │
       │  git push                      │                              │
       │  コードを「届ける」              │                              │
       │ ─────────────────────────────► │  リポジトリ（箱）に保存        │
       │                                │  ikedaharuki-1/               │
       │                                │  daily-mood-checkin           │
       │                                │                              │
       │                                │  デプロイのタイミングで         │
       │                                │  「最新の箱を取りに来る」      │
       │                                │ ─────────────────────────────►│
       │                                │                              │
       │                                │                    【ビルド】
       │                                │                    材料を組み立て
       │                                │                    pip install など
       │                                │                              │
       │                                │                    【起動】
       │                                │                    gunicorn でサーバー開始
       │                                │                              │
       │                                │                    【URL がもらえる】
       │                                │                    https://xxx.onrender.com
       │                                │                              │
```

---

## 3. 用語の関係（図）

```
  GitHub
    └── リポジトリ（1プロジェクト＝1箱）
            └── ブランチ（main など＝「どの版」か）
                    └── コミット（1回1回の保存の単位）

  デプロイ
    └── ① リポジトリからコードを取得
    └── ② ビルド（組み立て）
    └── ③ 起動して URL を公開
```

- **リポジトリ** … GitHub のなかにある「1プロジェクト分の入れ物」
- **ブランチ** … その入れ物のなかの「どのラインの版を使うか」。ふつうは **main** を使う
- **コミット** … 「この状態で保存」という1回分。push すると、そのコミットが GitHub に送られる
- **デプロイ** … 「取得 → ビルド → 起動」までまとめた流れ

---

## 4. ビルドとデプロイの違い（比喩）

| 用語 | やること（比喩） |
|------|------------------|
| **ビルド** | 材料（コード）を **「動く形に組み立てる」** だけ。まだ誰も使えない。例：`pip install -r requirements.txt` でライブラリを入れる。 |
| **デプロイ** | 組み立てたものを **「本番の場所に置いて、開店する」** まで。ビルドもその一部。＝ 「取りに来る → ビルド → 起動 → URL 公開」の一連の流れ。 |

だから **「デプロイした」＝「ビルドして、サーバーで動かして、URL を出した」** という意味になります。

---

## 5. 今回の流れ（まとめ）

1. **リポジトリ** … GitHub に「daily-mood-checkin」という**箱**（リポジトリ）を作った。
2. **push** … あなたのPCから、その箱に**コードを届けた**（git push）。
3. **Render でデプロイ** … Render が「その箱の最新の中身を取りにいく」と設定した。
4. **ビルド** … Render が取りに来たコードを **pip install** で組み立てた（ビルド）。
5. **起動** … **gunicorn** でサーバーを起動し、**URL** を付けた。
6. **デプロイ完了** … その URL で誰でもアプリにアクセスできる状態になった。

---

## 6. なぜビルドが必要？ ないとどうなる？

### 理由（一言）

**あなたのコードは「Flask」や「gunicorn」といった「他の人の書いたプログラム（ライブラリ）」に頼って動いている。**  
それらは GitHub のリポジトリには入っておらず、**requirements.txt に「これを使う」と書いてあるだけ**です。

- **ビルド** ＝ サーバー上で **「requirements.txt に書いてあるものをインストールする」** こと（＝ `pip install -r requirements.txt`）。
- これをやらないと、**「Flask がない」「gunicorn がない」** 状態のまま起動しようとするので、**動きません**。

### 比喩

- **あなたのコード** … レシピ（「Flask を使って API を書く」）
- **ライブラリ（Flask など）** … 材料（レシピに書いてある「買ってきて」の部分）
- **ビルド** … **材料を買ってきてそろえる**作業。  
  材料がなければ料理（＝アプリ）は作れない。

### ビルドがないとどうなるか

| やること | 結果 |
|----------|------|
| ビルドを **やる** | `pip install` で Flask や gunicorn がサーバーに入る → 起動コマンド（gunicorn）が実行できる → **アプリが動く**。 |
| ビルドを **やらない** | サーバーにはあなたの **app.py だけ**がある。Flask も gunicorn もない。  
  → `gunicorn app:app` と実行すると **「gunicorn が見つかりません」** や、Python が **「Flask を import できません」** というエラーで **起動に失敗**する。 |

だから **「ビルド＝動かす前に、必要なものをサーバーにそろえるステップ」** で、これがないとアプリは立ち上がりません。

# Render と同じようなサービス

**「コードを預けると、サーバーで動かして URL をくれる」** タイプのサービスは他にもあります。

---

## よく使われるもの（PaaS・ホスティング）

| サービス名 | 一言 | 特徴 |
|------------|------|------|
| **Render** | 今回使ったサービス | 無料枠あり。GitHub と連携してデプロイ。Python / Node / Docker など対応。 |
| **Railway** | 似た感じの PaaS | 無料枠あり（制限あり）。GitHub 連携。シンプルな画面。 |
| **Fly.io** | サーバーを世界に配置 | 無料枠あり。Docker ベース。複数リージョンにデプロイしやすい。 |
| **Heroku** | 昔からある PaaS | 無料枠は終了。有料なら安定して人気。 |
| **Vercel** | フロント・軽い API 向け | 無料枠あり。Next.js などフロント寄りが得意。サーバーless 関数も。 |
| **Netlify** | 静的サイト・フロント向け | 無料枠あり。HTML/JS の静的サイトや Jamstack 向け。関数も使える。 |
| **Google Cloud Run** | Google のサービス | コンテナを動かす。従量課金。無料枠あり。 |
| **AWS（Amazon Web Services）** | 大きなクラウド | EC2、Lambda、Elastic Beanstalk などいろいろある。学習コストは高め。 |

---

## 選び方の目安（感覚）

- **手軽に「URL だけ欲しい」** … Render / Railway / Fly.io  
- **フロント（React など）や静的サイト中心** … Vercel / Netlify  
- **企業・本番でがっつり** … Heroku 有料 / AWS / Google Cloud  

今回のように **Python（Flask）を 1 本デプロイして URL が欲しい** なら、**Render** や **Railway** が近いです。

---

*「GitHub＝倉庫、リポジトリ＝箱、push＝届ける、ビルド＝組み立て、デプロイ＝届けて組み立てて開店する」と覚えておくと、関係がつかみやすいです。*

---

## 料理の比喩でまとめると

| 用語 | 比喩 |
|------|------|
| **コード** | レシピ（「こういう材料で、こう作る」という設計） |
| **ビルド** | レシピに沿って**材料を集めてそろえる**（pip install で Flask などを入れる） |
| **起動** | 材料を使って**調理する**（gunicorn でアプリを動かす） |
| **デプロイ** | 材料をそろえて調理し、**お店に並べて出す**まで。＝ **レシピで完成した食事を、お客が食べられる状態にすること** |

だから「**コード＝レシピ、ビルド＝材料を集める、デプロイ＝完成した食事を出す**」という理解で合っています。
